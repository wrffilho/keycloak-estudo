# ADR-003: Stack Java Spring Boot com Monolito Modular Simples

## Status

Aceita

## Data

2026-05-10

## Contexto

A TechSpec precisa definir a primeira arquitetura do laboratorio, pois o repositorio ainda nao possui aplicacao, dependencias ou estrutura de codigo. O PRD exige uma API backend-first, sem frontend na V1, com Keycloak local, testes pelo Postman e possibilidade de evoluir para frontend depois.

O usuario escolheu Java + Spring Boot como stack principal e quer aproveitar o projeto para aprender monolito modular sem transformar uma V1 pequena em arquitetura excessiva.

## Decisao

A V1 sera implementada com Java + Spring Boot em um monolito modular simples. O projeto tera um backend principal agora, localizado em `backend/`, com pacotes internos organizados por responsabilidade e dominio, preservando um caminho claro para adicionar um frontend em fase posterior.

A estrutura inicial deve favorecer entendimento e evolucao:

- area de `documentos` para regras e endpoints do dominio;
- area de `seguranca` para autenticacao/autorizacao e conversao de permissoes;
- area de `configuracao` para configuracoes do Spring/Keycloak;
- area `compartilhado` apenas se houver tipos realmente reutilizados.

Na raiz, o `pom.xml` funciona como agregador Maven para permitir comandos de verificacao a partir do repositorio inteiro; o build real da API fica em `backend/pom.xml`.

## Alternativas Consideradas

### Alternativa 1: Estrutura por camadas tradicional

- **Descricao**: Organizar todo o codigo em pacotes como `controller`, `service`, `model`, `security` e `config`.
- **Prós**: Familiar para muitos projetos Spring e simples de navegar no inicio.
- **Contras**: Ensina menos sobre limites modulares e tende a espalhar funcionalidades por camadas.
- **Motivo da rejeicao**: O usuario quer aprender monolito modular, mesmo mantendo a V1 simples.

### Alternativa 2: Multimodulo completo desde a V1

- **Descricao**: Criar varios modulos fisicos desde o inicio, por exemplo backend, frontend futuro, seguranca, documentos e compartilhado.
- **Prós**: Limites mais explicitos e maior aproximacao com monolito modular formal.
- **Contras**: Aumenta configuracao e complexidade antes de existir necessidade real.
- **Motivo da rejeicao**: A V1 deve priorizar aprendizado de Keycloak e manter o projeto pequeno.

## Consequencias

### Positivas

- Mantem a implementacao simples para quem esta comecando do zero.
- Introduz pensamento modular sem burocracia excessiva.
- Deixa o frontend futuro como evolucao natural, sem exigir essa parte agora.
- Combina bem com Spring Boot e exemplos didaticos em PT-BR.

### Negativas

- Os limites modulares serao convencionais, nao impostos por build separado.
- Pode exigir disciplina na organizacao dos pacotes para nao virar um projeto por camadas disfarçado.

### Riscos

- **Excesso de pacotes cedo demais**: A estrutura pode ficar maior que o problema. Mitigacao: criar apenas pacotes usados pela V1.
- **Monolito modular superficial**: O projeto pode nomear modulos sem respeitar dependencias. Mitigacao: documentar as responsabilidades de cada area na TechSpec.

## Observacoes de Implementacao

Detalhes exatos de build, versoes e nomes finais ficam para a implementacao das tarefas. A TechSpec deve definir limites e responsabilidades, nao tentar antecipar todo o codigo.
