# ADR-001: Escopo da V1 para Laboratorio PT-BR de Autorizacao com Keycloak

## Status

Aceita

## Data

2026-05-10

## Contexto

O projeto sera um laboratorio de aprendizado para um desenvolvedor brasileiro que quer estudar Keycloak Authorization Services por meio de uma aplicacao demo executavel. O principal desafio de aprendizado e entender a diferenca entre endpoints publicos, endpoints autenticados e endpoints protegidos por autorizacao fina configurada com recursos, escopos, politicas, permissoes e URIs no Keycloak.

O repositorio esta praticamente vazio, entao a V1 pode definir a primeira forma do projeto. A pesquisa confirmou que o Keycloak Authorization Services protege recursos que podem ser mapeados para URIs, usa escopos para representar acoes, politicas para expressar regras de acesso e permissoes para conectar recursos/escopos a politicas. A demo precisa evitar ensinar atalhos inseguros, especialmente a ideia errada de que um token valido, por si so, significa que o usuario esta autorizado.

## Decisao

A V1 sera um laboratorio pequeno, executavel e em portugues do Brasil, centrado em um dominio simples, como `documentos` ou `relatorios`. Ela incluira uma API demo, Docker Compose para subir o Keycloak, configuracao do Keycloak e exemplos/documentacao para Postman que demonstrem de forma visivel quatro categorias de endpoints:

- endpoints publicos que nao exigem login;
- endpoints autenticados que exigem login valido, mas nao exigem permissao fina;
- endpoints protegidos por autorizacao que exigem aprovacao por recurso/escopo/politica/permissao no Keycloak;
- cenarios de acesso negado nos quais um usuario logado nao possui a permissao necessaria.

A postura padrao de autorizacao sera negar por padrao com enforcement habilitado no Keycloak. Os modos `permissive` e `disabled` podem ser documentados apenas como experimentos de aprendizado, nao como padroes recomendados.

O backend pode usar organizacao de monolito modular para permitir que o projeto receba um frontend depois sem reescrever os exemplos principais de autorizacao. Um frontend esta explicitamente fora do escopo da V1.

## Alternativas Consideradas

### Alternativa 1: Vitrine ampla de autorizacao

- **Descricao**: Cobrir muitos tipos de politica do Keycloak, varios dominios, muitos papeis e multiplos exemplos ja na V1.
- **Prós**: Cobertura mais completa do Keycloak Authorization Services.
- **Contras**: Maior carga cognitiva, setup mais longo e maior risco de o aprendiz ver uma demo funcionando sem entender cada decisao.
- **Motivo da rejeicao**: A V1 deve otimizar compreensao e conclusao antes de amplitude.

### Alternativa 2: Demo minima somente com autenticacao

- **Descricao**: Criar apenas endpoints publicos e autenticados, deixando exemplos de recurso/politica/permissao para depois.
- **Prós**: Mais rapido de construir e mais facil de rodar.
- **Contras**: Nao resolve o problema central de aprendizado: entender autorizacao fina por URI/recurso.
- **Motivo da rejeicao**: O objetivo explicito do usuario e aprender policies, permissions e resources com URI no Keycloak.

### Alternativa 3: Plataforma completa de estudos

- **Descricao**: Criar uma plataforma polida de estudo com frontend, exercicios, modulos, progresso e muitos cenarios.
- **Prós**: Experiencia de aprendizado mais forte no longo prazo.
- **Contras**: Transforma a V1 em um projeto de plataforma e atrasa o primeiro ciclo util de aprendizado.
- **Motivo da rejeicao**: O primeiro marco deve ser uma API executavel na qual cada requisicao permitida ou negada possa ser explicada.

### Alternativa 4: Demo assistida por frontend

- **Descricao**: Incluir um frontend simples na V1 para acionar endpoints publicos, autenticados e autorizados visualmente.
- **Prós**: Torna status codes e fluxos de login/logout mais visiveis.
- **Contras**: Adiciona setup de UI, tratamento de token, CORS, roteamento e depuracao extra antes de o modelo de autorizacao do Keycloak estar claro.
- **Motivo da rejeicao**: O usuario prefere testar primeiro com Postman e deixar o frontend para uma fase posterior.

## Consequencias

### Positivas

- O aprendiz consegue mapear cada endpoint para um recurso, escopo, politica e permissao visiveis no Keycloak.
- A nomenclatura em portugues do Brasil reduz carga cognitiva e deixa o projeto mais acessivel.
- Acesso negado vira um cenario de aprendizado de primeira classe, nao apenas uma falha acidental.
- O projeto pode ser concluido antes de adicionar conceitos mais amplos do Keycloak.
- Docker Compose torna o ponto de partida realista para quem esta com o ambiente local zerado.
- Limites de monolito modular preservam um caminho para um frontend futuro.

### Negativas

- A V1 nao cobrira todos os tipos de politica do Keycloak Authorization Services.
- A demo pode parecer menos realista que uma aplicacao de producao.
- Um laboratorio com dominio unico exige documentacao cuidadosa para que os conceitos sejam transferidos para outros sistemas.
- Testes via Postman exigem exemplos claros para tokens, status codes esperados e cenarios de falha.

### Riscos

- **Modelo mental inseguro**: O aprendiz pode assumir que autenticacao e a mesma coisa que autorizacao. Mitigacao: manter endpoints apenas autenticados e endpoints autorizados bem separados.
- **Permissoes amplas demais**: Padroes de URI podem ensinar acesso permissivo sem querer. Mitigacao: usar mapeamento explicito entre endpoint e recurso e usuarios de exemplo com menor privilegio possivel.
- **Crescimento de escopo**: Adicionar frontend, persistencia ou multiplos servicos pode esconder a licao de autorizacao. Mitigacao: deixar V1 focada em API, Keycloak, Docker e Postman.

## Observacoes de Implementacao

Usar nomes em portugues do Brasil sempre que fizer sentido para classes, funcoes, rotulos de endpoints, exemplos e documentacao. Manter os conceitos do Keycloak precisos com termos consistentes: `recurso`, `escopo`, `politica`, `permissao`, `autenticado`, `autorizado` e `negado`.

A V1 deve incluir Docker Compose para o Keycloak porque o usuario esta comecando do zero localmente. A API deve ser organizada em estilo de monolito modular se isso nao adicionar cerimonia desnecessaria.
