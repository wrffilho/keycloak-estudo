# ADR-005: Dados em Memoria e Testes Essenciais para V1

## Status

Aceita

## Data

2026-05-10

## Contexto

O PRD exige um laboratorio local simples, com foco em autorizacao por endpoints. O dominio `documentos` existe para sustentar cenarios de permissao, nao para criar um sistema documental real. O usuario escolheu dados em memoria e testes essenciais para manter o foco no aprendizado.

## Decisao

A V1 usara dados de `documentos` em memoria, com uma colecao pequena e previsivel de registros. Os testes automatizados devem cobrir o essencial:

- endpoint publico retorna sucesso sem token;
- endpoint autenticado retorna `401` sem token;
- endpoint autenticado retorna sucesso com token valido;
- endpoint autorizado retorna `403` quando o usuario nao possui permissao;
- endpoint autorizado retorna sucesso quando o usuario possui permissao.

A validacao manual pelo Postman continua sendo parte central da experiencia de aprendizado, mas nao substitui os testes essenciais da API.

## Alternativas Consideradas

### Alternativa 1: Banco de dados via Docker

- **Descricao**: Incluir um banco de dados local no Docker Compose para armazenar documentos.
- **Prós**: Mais realista e prepara melhor para uma aplicacao completa.
- **Contras**: Adiciona setup e problemas que nao ajudam a aprender autorizacao na V1.
- **Motivo da rejeicao**: Persistencia nao e necessaria para demonstrar recursos, escopos, politicas e permissoes.

### Alternativa 2: Apenas testes manuais pelo Postman

- **Descricao**: Validar todos os cenarios manualmente, sem testes automatizados.
- **Prós**: Menor esforco inicial e mais foco na exploracao manual.
- **Contras**: Dificulta detectar regressao nos cenarios centrais.
- **Motivo da rejeicao**: Testes essenciais dao confianca sem transformar a V1 em suite completa.

## Consequencias

### Positivas

- Reduz a complexidade operacional da V1.
- Mantem foco na diferenca entre `200`, `401` e `403`.
- Facilita repetir cenarios com dados previsiveis.
- Cria uma rede minima de seguranca para os comportamentos mais importantes.

### Negativas

- O laboratorio nao demonstra persistencia real.
- Alguns cenarios de negocio ficam simplificados.
- Testes de integracao com Keycloak podem exigir cuidado adicional na implementacao.

### Riscos

- **Dados parecerem artificiais demais**: Mitigacao: usar exemplos de documentos realistas, mas simples.
- **Testes essenciais virarem suite complexa**: Mitigacao: limitar V1 aos cenarios de status HTTP e permissoes principais.

## Observacoes de Implementacao

Os testes devem priorizar clareza sobre cobertura exaustiva. A TechSpec deve deixar os cenarios de teste alinhados ao roteiro do Postman.
