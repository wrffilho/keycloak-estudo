# ADR-005: Composable Simples E Rotas Hibridas No Frontend

## Status

Accepted

## Date

2026-05-11

## Context

O frontend atual usa Vue, Vue Router e composables simples para sessao, permissoes, documentos e diagnostico. Nao ha store global como Pinia.

O PRD exige rotas reais para menus ficticios, mas nao exige telas completas para cada modulo. Tambem exige que menus importantes existentes continuem funcionando.

## Decision

O frontend usara um composable simples, como `useMenuDinamico`, para carregar, armazenar e expor o menu dinamico.

As rotas seguirao uma abordagem hibrida:

- rotas reais ja existentes ou importantes continuam explicitas;
- menus ficticios usam uma rota generica compartilhada para tela de confirmacao de acesso.

O guard de rota usara o estado do menu dinamico para decidir se uma rota ficticia acessada diretamente esta liberada para o usuario. Se nao estiver, exibira alerta fixo de acesso negado e redirecionara para `/laboratorio` ao fechar ou confirmar.

## Alternatives Considered

### Alternative 1: Store global

- **Description**: Adicionar Pinia ou outra store para controlar menu, sessao e permissoes.
- **Pros**: Pode ser util em aplicacoes maiores.
- **Cons**: Adiciona dependencia e complexidade que o projeto ainda nao usa.
- **Why rejected**: O padrao atual de composables resolve a V1 com menos mudanca.

### Alternative 2: Estado apenas no cabecalho

- **Description**: Carregar e guardar o menu diretamente em `CabecalhoPrincipal.vue`.
- **Pros**: Menos arquivos.
- **Cons**: O guard de rota e outras telas nao conseguiriam reutilizar o estado com clareza.
- **Why rejected**: O estado do menu precisa servir ao cabecalho e ao controle de rota.

### Alternative 3: Todas as rotas explicitas

- **Description**: Cadastrar cada rota ficticia individualmente.
- **Pros**: Mais proximo de um sistema real completo.
- **Cons**: Aumenta repeticao sem necessidade, ja que as telas ficticias compartilham o mesmo comportamento.
- **Why rejected**: A V1 precisa simular navegacao real sem implementar um portal completo.

## Consequences

### Positive

- Mantem o frontend alinhado aos padroes existentes.
- Evita dependencia nova.
- Permite navegar para telas ficticias reais sem criar muitos componentes.

### Negative

- O composable precisa ser carregado cedo o suficiente para o guard de rota.
- A rota generica exige cuidado para exibir o rotulo correto da tela.

### Risks

- **Estado ainda nao carregado durante acesso direto**: mitigar carregando menu antes de validar rotas protegidas.
- **Rota generica aceitar item inexistente**: mitigar validando o identificador contra a arvore de menu recebida.

## Implementation Notes

- O TechSpec deve definir o contrato de `ItemDeMenu`.
- O alerta de acesso negado deve seguir estilo visual existente e permanecer fixo ate o usuario fechar ou confirmar.
