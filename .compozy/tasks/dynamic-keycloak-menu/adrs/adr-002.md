# ADR-002: Menu Corporativo Didatico Com Controle De Rota

## Status

Accepted

## Date

2026-05-11

## Context

O PRD precisa transformar a ideia de menu dinamico em uma experiencia demonstravel. O objetivo nao e apenas provar que um JSON de menu pode ser carregado, mas sim mostrar um cenario parecido com um sistema corporativo maior, com menus, submenus, usuarios diferentes e rotas reais no Vue Router.

O projeto ja possui usuarios reais do laboratorio (`leitor`, `editor` e `aprovador`), mas eles cobrem principalmente o dominio de documentos. Para demonstrar menus como Cadastros, Operacoes, Relatorios e Administracao, a V1 precisa incluir alguns usuarios/perfis ficticios adicionais.

Tambem foi definido que rotas ficticias devem funcionar como navegacao real. Quando o usuario tem acesso, a tela mostra uma mensagem didatica. Quando nao tem acesso e tenta abrir a rota diretamente, a aplicacao deve mostrar um alerta fixo de acesso negado e redirecionar para a home ao fechar ou confirmar.

## Decision

A V1 adotara a abordagem de menu corporativo didatico com controle de rota.

O produto deve manter os usuarios atuais e adicionar 1 ou 2 usuarios ficticios para demonstrar combinacoes de acesso de um sistema maior. O menu deve incluir pelo menos os grupos Cadastros, Operacoes, Relatorios e Administracao, com submenus suficientes para testar hierarquia, ordenacao e diferencas de visibilidade.

As rotas ficticias devem existir no Vue Router e apontar para uma pagina compartilhada simples. Quando o acesso for permitido, a pagina exibira uma mensagem como `Usuario {nome} acessou {rotulo} porque o menu foi liberado pelo Keycloak.` Quando o acesso nao for permitido por acesso direto a URL, a experiencia deve mostrar um alerta fixo estilo Bootstrap com mensagem de acesso negado e redirecionar para `/laboratorio` quando o usuario fechar ou confirmar.

## Alternatives Considered

### Alternative 1: Menu minimo so para provar o fluxo

- **Description**: Manter poucos itens, quase todos ligados a Laboratorio e Documentos.
- **Pros**: Mais rapido e menor.
- **Cons**: Testa pouco submenu, hierarquia, ordenacao e diferencas entre perfis.
- **Why rejected**: Nao atende bem ao objetivo de simular um sistema maior.

### Alternative 2: Portal corporativo mais completo

- **Description**: Criar muitos perfis, menus e combinacoes detalhadas.
- **Pros**: Demonstra uma experiencia visual mais rica.
- **Cons**: Aumenta muito o escopo e pode atrasar o aprendizado central sobre Keycloak e menu dinamico.
- **Why rejected**: A V1 deve ser realista, mas ainda pequena o suficiente para executar e aprender.

## Consequences

### Positive

- A demonstracao fica mais proxima de um sistema corporativo real.
- O projeto passa a validar menus, submenus, rotas e acesso direto por URL.
- Usuarios diferentes conseguem demonstrar arvores de navegacao diferentes.
- O frontend pode simular telas reais sem implementar dominios completos.

### Negative

- A V1 precisara de mais massa de dados no realm.
- O produto tera rotas ficticias que precisam ser claramente apresentadas como demonstrativas.
- O escopo de validacao aumenta em relacao a um menu minimo.

### Risks

- **Usuario confundir rota ficticia com funcionalidade real**: mitigar com mensagem didatica clara na pagina compartilhada.
- **Escopo crescer demais**: limitar a V1 a poucos grupos e submenus representativos.
- **Experiencia de acesso negado ficar frustrante**: manter mensagem objetiva e redirecionamento previsivel para `/laboratorio`.

## Implementation Notes

- O PRD deve permanecer focado no comportamento esperado, sem definir detalhes de API ou estrutura de codigo.
- O TechSpec definira o formato final dos itens de menu, a estrategia de rota e a integracao exata com Keycloak.
