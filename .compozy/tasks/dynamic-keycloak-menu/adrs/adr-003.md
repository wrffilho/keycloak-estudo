# ADR-003: Catalogo Hibrido Com Resources E Roles No Client De Menu

## Status

Accepted

## Date

2026-05-11

## Context

O menu dinamico precisa ser cadastrado no Keycloak sem confundir rota de frontend com permissao de endpoint backend. A V1 tambem precisa ser didatica para quem esta aprendendo Keycloak.

O client `laboratorio-menu` sera dedicado ao catalogo de navegacao. A decisao tecnica pendente era como representar itens de menu e como liberar sua visibilidade para usuarios.

## Decision

O client `laboratorio-menu` usara um modelo hibrido:

- resources/attributes representam o catalogo de menu e submenu;
- roles do client `laboratorio-menu` liberam a visibilidade dos itens para usuarios.

Cada resource de menu deve conter metadados como rota, rotulo, ordem, icone, tipo e relacao pai/filho. Cada item tambem deve indicar a role necessaria para aparecer no menu do usuario.

## Alternatives Considered

### Alternative 1: Apenas roles do client de menu

- **Description**: Usar roles para representar diretamente cada item de menu.
- **Pros**: Simples para liberar acesso.
- **Cons**: Roles nao modelam bem metadados como rota, ordem, icone e hierarquia.
- **Why rejected**: O catalogo de navegacao precisa de atributos ricos alem da permissao de visibilidade.

### Alternative 2: Authorization Services completo

- **Description**: Usar resources, scopes, policies e permissions para toda decisao de visibilidade.
- **Pros**: Mais alinhado ao modelo completo de autorizacao fina do Keycloak.
- **Cons**: Aumenta complexidade para a V1 e pode dificultar o objetivo didatico inicial.
- **Why rejected**: A V1 precisa ensinar o fluxo sem exigir todo o modelo de Authorization Services de uma vez.

## Consequences

### Positive

- Separa catalogo visual de regra de visibilidade.
- Mantem a V1 compreensivel para estudo.
- Permite menus com hierarquia e metadados sem hardcode no frontend.

### Negative

- Exige convencao clara para relacionar resource de menu e role necessaria.
- Ainda nao explora todo o potencial de policies/permissions do Authorization Services.

### Risks

- **Convencao ficar inconsistente**: mitigar documentando atributos obrigatorios no TechSpec.
- **Roles crescerem demais**: limitar a V1 a poucos menus representativos.

## Implementation Notes

- O TechSpec deve definir os atributos obrigatorios do resource de menu.
- A avaliacao de visibilidade deve usar roles efetivas do usuario no client `laboratorio-menu`.
