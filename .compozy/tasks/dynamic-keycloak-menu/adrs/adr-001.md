# ADR-001: Menu Dinamico A Partir Do Client Frontend No Keycloak

## Status

Accepted

## Date

2026-05-11

## Context

O projeto e um laboratorio Keycloak em portugues do Brasil com backend Spring Boot, frontend Vue e dois clients principais no realm: um client para a API/backend e outro client para a SPA/frontend.

A ideia e demonstrar um menu dinamico em que o usuario loga no frontend, o frontend chama um endpoint autenticado do backend, e o backend consulta o Keycloak para descobrir quais rotas de menu e submenu devem ser disponibilizadas para aquele usuario. O frontend nao deve montar esse menu a partir de um endpoint generico de permissoes nem interpretar diretamente claims do token.

Ha uma distincao importante entre rota de frontend e endpoint de backend. Uma tela `/x` pode chamar varios endpoints, como `/y`, `/z` e `/w`, e cada endpoint continua tendo sua propria autorizacao. Portanto, o menu nao deve ser modelado como um simples espelho das permissoes de API.

## Decision

A V1 deve manter tres clients no Keycloak:

- um client de backend/resource server, responsavel por proteger APIs e validar permissoes de endpoints;
- um client de frontend, responsavel pela autenticacao da SPA;
- um client confidencial dedicado ao catalogo de menu/submenu do frontend, responsavel por armazenar os recursos de navegacao e suas regras de visibilidade.

O backend tera um endpoint autenticado, como `GET /usuario/menu`, chamado apos o login. Esse endpoint usara uma credencial de servico ou integracao administrativa apropriada para acessar o Keycloak, consultar o client confidencial de menu, obter os recursos/metadados de menu e submenu, filtrar o que se aplica ao usuario logado e devolver uma arvore JSON pronta para o Vue.

O frontend recebera esse JSON, ajustara somente detalhes de renderizacao quando necessario e exibira a navegacao dinamica. O frontend nao deve conhecer a regra administrativa do Keycloak nem decidir sozinho quais rotas existem para cada usuario.

Na V1, os menus/submenus devem ser cadastrados no client confidencial de menu usando recursos do Authorization Services ou metadados equivalentes, com campos como rota, rotulo, ordem, icone, tipo e relacao pai/filho. O objetivo e aprender a modelar o catalogo de navegacao de forma correta, sem sobrecarregar o client publico da SPA nem misturar o catalogo de menu com os recursos de API.

## Alternatives Considered

### Alternative 1: Usar `/usuario/permissoes` para montar o menu

- **Description**: O frontend ou backend usaria permissoes efetivas como `documentos:ler` para decidir quais links aparecem.
- **Pros**: Reaproveita estruturas ja existentes no laboratorio.
- **Cons**: Confunde permissao de endpoint com rota de frontend e nao cobre telas que carregam multiplos recursos internos.
- **Why rejected**: O objetivo e testar menu/submenu definido no Keycloak para a experiencia do frontend, nao apenas ocultar links por permissoes de API.

### Alternative 2: Menu fixo no frontend filtrado por claims do token

- **Description**: O frontend manteria todas as rotas e filtraria com base em roles, groups ou claims.
- **Pros**: Simples de implementar inicialmente.
- **Cons**: Acopla a SPA ao formato do token, espalha regra de autorizacao visual e dificulta mudar a modelagem no Keycloak.
- **Why rejected**: O backend deve ser o adaptador entre Keycloak e frontend, entregando um contrato de menu limpo.

### Alternative 3: Mapeamento local de menu no backend

- **Description**: O backend manteria uma tabela local de item de menu -> permissao.
- **Pros**: Facil de testar e controlar no codigo.
- **Cons**: Nao valida a hipotese principal de cadastrar rotas de menu/submenu no Keycloak.
- **Why rejected**: Pode ser fallback tecnico, mas nao e a direcao principal da V1.

### Alternative 4: Cadastrar menus no client `laboratorio-api`

- **Description**: O client de API/resource server guardaria tanto recursos de backend quanto recursos de menu frontend.
- **Pros**: Evita criar outro client e aproveita infraestrutura existente.
- **Cons**: Mistura responsabilidades diferentes e reduz o valor didatico de aprender a separacao correta entre login, API e catalogo de navegacao.
- **Why rejected**: A V1 deve priorizar aprendizado correto e separacao clara, criando um client confidencial dedicado ao menu.

## Consequences

### Positive

- O Keycloak se torna a fonte administrativa do catalogo de navegacao do frontend.
- O projeto ensina a separacao entre client de login, client de API e client de catalogo de navegacao.
- O frontend recebe um contrato simples e nao precisa entender detalhes de roles, resources, scopes ou Admin API.
- A solucao preserva a separacao entre menu de navegacao e autorizacao real de endpoints.
- O laboratorio passa a demonstrar um cenario mais proximo de sistemas corporativos com menu dinamico.

### Negative

- A V1 fica mais investigativa e depende de entender bem as APIs administrativas/autorizacao do Keycloak.
- O backend passa a precisar de credencial para consultar o Keycloak, aumentando configuracao e superficie operacional.
- A configuracao do realm fica maior porque passa a ter um client adicional dedicado ao menu.

### Risks

- **Confundir menu com seguranca**: mitigar deixando claro que endpoints continuam protegidos independentemente do menu.
- **Uso excessivo de privilegios administrativos**: mitigar com service account de menor privilegio possivel e acesso restrito ao necessario.
- **Acoplamento a detalhes internos do Keycloak**: mitigar isolando a integracao em um servico backend dedicado.
- **Metadados insuficientes no Keycloak**: mitigar com uma prova tecnica antes de fechar o TechSpec.

## Implementation Notes

- Criar um contrato de resposta em PT-BR para a arvore de menu, por exemplo `ItemDeMenuResposta`.
- Criar um servico backend dedicado, por exemplo `ServicoDeMenuDoKeycloak`, isolando chamadas ao Keycloak.
- Criar um client confidencial dedicado, por exemplo `laboratorio-menu`, para representar o catalogo de navegacao do frontend.
- Avaliar o uso de resources/attributes do Authorization Services nesse client para representar itens como `menu.documentos`, `submenu.documentos.aprovacao` e rotas como `/documentos`.
- Incluir campos esperados: `id`, `rotulo`, `rota`, `tipo`, `ordem`, `icone`, `pai` e `filhos`.
- O frontend deve chamar `GET /usuario/menu` apos login antes de renderizar a navegacao principal.
- O frontend deve continuar tratando `401` e `403` das APIs reais como resultado de autorizacao do backend, nao do menu.
