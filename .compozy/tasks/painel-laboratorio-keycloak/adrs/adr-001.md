# ADR-001: Escopo da V1 como Painel Didatico de Autorizacao

## Status

Accepted

## Date

2026-05-11

## Context

O laboratorio atual ensina autorizacao com Keycloak por meio de backend Spring Boot, realm importado, Postman e documentacao em portugues do Brasil. O frontend proposto deve evoluir esse aprendizado sem abandonar a maxima do projeto: reduzir carga cognitiva usando nomes, telas, exemplos e explicacoes em PT-BR.

A ideia refinada combina tres expectativas: uma demo apresentavel, um painel de laboratorio para visualizar token/permissoes/respostas HTTP e um pouco de portal de documentos/admin simples. O risco e transformar a V1 em um produto grande demais, com CRUD elaborado, UX administrativa e detalhes de OAuth competindo pela atencao do aprendiz.

O council convergiu que o valor central nao e criar um gerenciador documental completo. O valor central e tornar observavel a cadeia: usuario -> token -> permissoes -> endpoint -> resposta HTTP.

## Decision

A V1 do frontend sera um Painel do Laboratorio Keycloak, em portugues do Brasil, com login real via Keycloak, client SPA separado, operacoes essenciais de documentos e diagnostico explicito de autorizacao.

O painel deve permitir listar, ver detalhe, criar, editar e aprovar documentos, mas apenas no nivel necessario para exercitar permissoes. Cada acao protegida deve deixar visivel a permissao exigida, as permissoes do usuario, o endpoint chamado, o metodo HTTP e a resposta retornada (`200`, `201`, `401`, `403` ou erro aplicavel).

A V1 nao sera um admin completo, nem um gerenciador documental rico. A interface deve ser apresentavel, mas sempre didatica.

## Alternatives Considered

### Alternative 1: Portal de documentos completo

- **Description**: Construir uma aplicacao com foco principal em gestao documental, com CRUD polido, filtros, estados, detalhes ricos e fluxo de aprovacao mais realista.
- **Pros**: Mais parecida com um produto real e mais facil de demonstrar como aplicacao de negocio.
- **Cons**: Aumenta complexidade de frontend e pode esconder os conceitos de autorizacao atras de regras de negocio e UI.
- **Why rejected**: O projeto e um laboratorio de autorizacao, nao um sistema documental. O escopo desviaria do aprendizado.

### Alternative 2: Console tecnico de API

- **Description**: Criar uma tela parecida com Postman, focada em escolher endpoint, enviar requisicao e ver resposta.
- **Pros**: Simples, rapido e direto para testar endpoints.
- **Cons**: Pouco apresentavel como demo de aplicacao real e menos natural para mostrar permissoes no contexto do dominio `documentos`.
- **Why rejected**: O usuario quer uma demo completa, com um pouco de portal e admin simples, nao apenas um cliente HTTP.

### Alternative 3: Painel didatico de laboratorio

- **Description**: Combinar uma area de documentos realista o suficiente com um painel de diagnostico que mostra usuario, permissoes, endpoint, metodo e status.
- **Pros**: Equilibra aprendizado, teste rapido e demo apresentavel sem esconder autorizacao.
- **Cons**: Exige disciplina de escopo para nao virar portal/admin completo.
- **Why rejected**: Nao rejeitada. Esta e a decisao adotada.

## Consequences

### Positive

- Mantem a maxima PT-BR-first do projeto.
- Torna visivel a diferenca entre autenticacao e autorizacao.
- Permite demonstrar o backend sem depender exclusivamente do Postman.
- Preserva o dominio `documentos` como fio condutor do aprendizado.
- Evita ensinar habitos inseguros, como expor client secret no navegador.

### Negative

- A V1 tera menos funcionalidades de produto documental do que um portal real.
- O frontend precisara de configuracao adicional no realm para um client SPA publico.
- O painel de diagnostico adiciona trabalho de UI que nao existiria em um CRUD simples.

### Risks

- **Risco: esconder autorizacao na UI**. Mitigacao: sempre exibir permissao exigida, resposta HTTP e explicacao curta de `401`/`403`.
- **Risco: ensinar OAuth inseguro**. Mitigacao: usar client SPA separado, publico, com Authorization Code + PKCE e redirect URIs especificas.
- **Risco: escopo crescer demais**. Mitigacao: manter fora da V1 filtros avancados, admin de usuarios, edicao de realm e historico persistente.

## Implementation Notes

- Criar ou ajustar realm com um client SPA separado do `laboratorio-api`.
- O client SPA deve ser publico e nao deve carregar secret no navegador.
- O painel deve usar nomes em portugues do Brasil: `Documentos`, `Permissoes`, `Usuario`, `Acesso negado`, `Requisicao`, `Resposta`.
- O backend continua sendo a fonte real da autorizacao. O frontend pode desabilitar ou explicar botoes, mas nao substitui validacao no backend.
