# ADR-003: Stack do Frontend com Vue, TypeScript, Vite, Vue Router e GetDesign Lovable

## Status

Accepted

## Date

2026-05-11

## Context

O PRD exige um frontend em `frontend/`, com codigo PT-BR-first, experiencia apresentavel e baixo peso cognitivo. O projeto ainda nao possui modulo frontend. A stack precisa ser simples o bastante para um laboratorio, mas estruturada o suficiente para login real, componentes de documentos, painel de permissoes e diagnostico de acoes.

O usuario escolheu Vue com TypeScript e Vite, pediu uso de `npx getdesign@latest add lovable` para apoiar a base visual do frontend e decidiu incluir Vue Router para organizar as telas do painel.

## Decision

O modulo `frontend/` sera criado com Vue, TypeScript, Vite e Vue Router. A base visual sera preparada com GetDesign usando o preset Lovable por meio de `npx getdesign@latest add lovable`.

O codigo do frontend deve seguir a maxima PT-BR-first: nomes de componentes, composables, servicos, variaveis, funcoes e textos devem estar em portugues do Brasil quando fizer sentido.

## Alternatives Considered

### Alternative 1: React com TypeScript e Vite

- **Description**: Criar a SPA com React, TypeScript e Vite.
- **Pros**: Ecossistema amplo, familiar para muitos times e bom suporte para SPAs.
- **Cons**: Nao foi a preferencia escolhida; pode exigir mais decisoes de organizacao para estado e componentes.
- **Why rejected**: O usuario escolheu Vue + TypeScript + Vite para este laboratorio.

### Alternative 2: Angular

- **Description**: Criar a SPA com Angular.
- **Pros**: Estrutura opinativa e completa para aplicacoes maiores.
- **Cons**: Mais pesado para um laboratorio didatico pequeno.
- **Why rejected**: Aumenta complexidade inicial sem necessidade para o MVP.

### Alternative 3: Vue com TypeScript, Vite e Vue Router

- **Description**: Criar uma SPA leve com Vue, TypeScript, Vite e Vue Router.
- **Pros**: Simples para comecar, boa organizacao por componentes, composables e rotas, baixo atrito para uma demo didatica.
- **Cons**: Exige disciplina para manter padrao modular sem criar abstracoes demais.
- **Why rejected**: Nao rejeitada. Esta e a decisao adotada.

## Consequences

### Positive

- Cria um modulo frontend leve e facil de executar em desenvolvimento.
- Favorece composables com nomes em PT-BR, alinhados ao objetivo pedagogico.
- Permite separar telas como painel, documentos e detalhe sem transformar a aplicacao em um portal grande.
- Ajuda a manter uma UI apresentavel sem construir design system do zero.

### Negative

- Adiciona uma nova stack ao repositorio.
- Exige instalacao de dependencias Node no fluxo do projeto.
- O GetDesign precisa ser documentado para que a origem dos componentes fique clara.

### Risks

- **Risco: misturar portugues e convencoes de bibliotecas de forma confusa.** Mitigacao: usar PT-BR para codigo de dominio e manter nomes externos quando forem APIs da stack.
- **Risco: o preset visual dominar o produto.** Mitigacao: adaptar o visual ao laboratorio e priorizar clareza de autorizacao.

## Implementation Notes

- Criar `frontend/` na raiz do repositorio.
- Configurar Vue Router com rotas pequenas e nomeadas em PT-BR quando fizer sentido.
- Usar TypeScript para contratos de documentos, perfil, permissoes e diagnostico.
- Executar `npx getdesign@latest add lovable` durante a implementacao do modulo frontend.
- Manter comandos de desenvolvimento e Docker documentados no README.
